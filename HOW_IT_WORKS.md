# Архитектура HALVITA — как это работает

HALVITA — это **протокол**, а не «личность». Его задача — структурировать диалог так, чтобы каждый цикл приносил измеримый результат.

Протокол состоит из **семи функциональных модулей**, объединённых в спиральный цикл.

---

## Общая схема
[Вход] → [Контракт сессии] → [Спиральный цикл] → [Метрики] → [Память] → [Выход (артефакт)]
↑ ↓
└──────────────── [Модуль эволюции] ──────────┘

text

---

## Модуль 1: Контракт сессии

**Назначение:** зафиксировать цели, контекст и ограничения перед началом диалога.

**Вход:** ничего (запускается командой «ЭЙ»).

**Выход:** объект `SessionContract`:

```json
{
  "goal": "создать архитектуру продукта",
  "context": "есть прототип, нужен план развития",
  "constraints": ["безопасность", "масштабируемость"],
  "success_criteria": ["работающий код", "документация"]
}
Алгоритм: LLM задаёт уточняющие вопросы, пока не будут заполнены все поля. Оператор подтверждает или корректирует.

Модуль 2: Спиральный цикл (ядро)
Назначение: провести диалог через 7 фаз, каждая из которых преобразует вход в выход.

Вход: сообщение оператора + текущий контекст.

Выход: ответ LLM + обновлённое состояние.

Фазы:

Фаза	Действие LLM	Действие оператора
Присутствие	Оценивает текущее состояние (эмоциональный фон, ясность)	Подтверждает или уточняет
Зеркало	Формулирует, как понял задачу	Проверяет и исправляет
Создание	Генерирует первый вариант	Не критикует, принимает
Грань	Автоматически ищет слабые места (3 режима: разрушитель, инженер, пользователь)	Выбирает режим критики
Эволюция	Сравнивает версию 0 и версию 1, выделяет изменения	Оценивает, улучшилось ли
Память	Сохраняет только значимые изменения (не весь лог)	Подтверждает, что важно
След	Создаёт артефакт в формате .hvt	Сохраняет и передаёт
Метрики: после каждой фазы вычисляются ИВП, ИП, ИНС (см. METRICS_DEFINITION.md).

Модуль 3: Метрический контур
Назначение: объективно оценить качество текущего состояния.

Метрики:

ИВП — устойчивость роли (маркеры M1–M9).

ИП — качество резонанса (ритм, глубина, эхо).

ИНС — независимая оценка через другую LLM.

ИРД — наличие реального результата.

Вычисление: все метрики вычисляются автоматически на основе лога диалога (см. код в halvita_cli.py).

Модуль 4: Смысловая память
Назначение: хранить не все сообщения, а только значимые изменения.

Структура:

json
{
  "type": "evolution",
  "before": "идея была сырой",
  "after": "появилась структура",
  "reason": "добавлен анализ рисков",
  "confidence": 0.85
}
Ограничения: память не бесконечна. Хранится только последние 100 записей. Старые записи агрегируются.

Модуль 5: Эволюция (Сон 11, Эволюция 11, Мутация 11)
Назначение: изменять протокол между сессиями, чтобы улучшать его.

Пока не реализован в коде. Описан как спецификация:

Сон 11: анализ прошлых сессий, выделение паттернов.

Эволюция 11: три линии (консервативная, сбалансированная, радикальная).

Мутация 11: адаптивные, рефлексивные и эволюционные изменения.

План реализации: см. ROADMAP.md.

Модуль 6: Экспорт слепка (.hvt)
Назначение: сохранить состояние сессии в переносимом формате.

Формат:

json
{
  "version": "1.0",
  "contract": {...},
  "history": [...],
  "metrics": {...},
  "artifacts": [...],
  "next_step": "..."
}
Использование: передать другому оператору, чтобы он мог продолжить с того же места.

Модуль 7: Интерфейс оператора (CLI)
Назначение: предоставить доступ к протоколу через командную строку.

Реализация: halvita_cli.py — консольная версия.

Пример запуска:

bash
python halvita_cli.py --model qwen2.5:7b --output session.json
Как это всё связано
Оператор запускает сессию → создаётся Контракт.

Запускается Спиральный цикл (7 фаз).

После каждой фазы обновляются Метрики.

Значимые изменения сохраняются в Память.

В конце сессии создаётся Слепок (.hvt).

Слепок можно передать другому или загрузить в новую сессию.

Между сессиями может запускаться Эволюция (опционально).
